# Roouty 명세 + MCP 결합 분석 (roouty-spec)

Roouty 기능 관련 작업(**TC 작성·기획서 리뷰·이슈 분석** 등)을 할 때, 추측하지 않고
이 스킬의 `docs/` 제품 명세를 **근거(ground truth)** 로 삼는다. 그 위에 **MCP로 가져온 실제
티켓·기획·디자인**을 결합해, 일반론이 아니라 **실제 루티 서비스에 밀착된 산출물**을 만든다.

핵심 원칙:
- **명세(`docs/`)** = 안정적인 "정답지". 화면의 정상 동작·권한·검색·목록·예외 규칙.
- **MCP** = 변하는 "실시간 입력". 이번에 무엇을 만들/고치/검증하는지(티켓·기획·디자인).
- 두 소스를 **대조**해 차이·누락 AC·권한 케이스·엣지를 도출하는 것이 이 스킬의 목적이다.

## 적용 범위 (중요)

- 이 스킬의 명세는 **Roouty PC 웹 어드민**(자동배차·배차계획·모니터링·설정 등, 라우트 `/manage/...`·`/setting`) 전용이다.
- **기사앱(모바일/드라이버 앱) TC·분석은 이 스킬이 아니라 기사앱 전용 스킬·문서를 사용**한다. 대상이 모바일 기사앱이면 이 스킬을 적용하지 말고 그쪽으로 넘긴다.
- 웹/모바일이 섞인 티켓이면 **웹 화면 부분만** 이 스킬로 다룬다.

## 동작 절차

### 1단계 — 입력 식별 & 티켓 우선 조회
1. 입력에 **Jira 티켓 키(예: TECH-1234)나 URL이 있으면, 다른 무엇보다 먼저** `mcp__atlassian__getJiraIssue` 로 조회한다.
   - 티켓의 **컴포넌트·라벨·제목·본문·AC**에서 대상 화면 신호를 뽑는다. (이게 라우팅의 1차 근거)
2. 티켓이 없으면 요청 문장의 화면명·라우트·키워드에서 대상 화면을 식별한다.
3. 대상이 **모바일 기사앱**으로 판단되면 위 "적용 범위"에 따라 중단하고 기사앱 흐름으로 넘긴다.

### 2단계 — INDEX 자동 라우팅 → 명세 로드
1. 1단계에서 얻은 신호(컴포넌트/제목/라우트/키워드)로 **[INDEX.md](INDEX.md)** 화면맵을 매칭한다.
   - **사용자에게 "어느 문서 볼까요"라고 되묻지 않는다.** 신호로 자동 선택한다. 후보가 둘 이상이면 가장 가까운 1~3개를 모두 연다.
2. 매칭된 **`docs/<파일명>` 을 Read** 한다. (전체를 다 읽지 말고 관련 1~3개만)
   - 권한·노출 조건이 얽힌 기능이면 **`docs/00.메뉴구조(권한).md`** 를 함께 읽는다.
   - 도메인 용어/전체 흐름이 필요하면 **`docs/00.비즈니스요약.md`** 를 참고한다.

### 3단계 — 나머지 MCP 입력 수집 (필요 시)
- **Confluence** (`mcp__atlassian__getConfluencePage` 등): 기획서 원문. 파이프라인은 URL이 입력으로 들어온다.
- **Figma** (`mcp__claude_ai_Figma__*`): 화면/컴포넌트 디자인. 명세에 이미지가 없으므로 화면 확인은 여기서.
- cloudId 등 환경값은 시스템 프롬프트의 Atlassian 설정을 따른다. 불필요한 소스는 건너뛴다.

### 4단계 — 대조 (명세 vs 실제 요구)
- 명세의 **"정상 동작·권한·검색 필드·목록 컬럼·실패/예외"** 와 티켓/기획의 **요구사항**을 나란히 둔다.
- **차이**를 도출한다: 티켓에 없지만 명세엔 있는 권한·엣지 케이스 / 명세에 없는 신규 요구 / 둘이 모순되는 지점.

### 5단계 — 산출물 생성 + 근거 명시
- 본래 작업(TC/리뷰/분석)을 수행한다.
- 결과물에 **근거 3종을 명시**한다:
  `근거: 명세 docs/77.설정-납품처관리.md + Jira TECH-1234 + Figma <노드/프레임명>`
- 명세에 **없거나 모순되는 부분**은 임의로 메우지 말고 **"명세 미정의"** 로 별도 표시한다.

## 작업 유형별 활용

- **테스트 케이스 작성**: 명세의 `검색 필드`·`목록 컬럼`·`탭별 노출 조건`·`권한 분기`·`Actions의 실패/예외`를
  그대로 정상/예외/권한/경계 TC로 전환한다. 권한·탭별 노출표는 매트릭스 TC의 원천. 여기에 Jira AC를
  더해 이번 변경분 TC를 보강한다.
- **기획서/티켓 리뷰**: 티켓 AC를 명세와 대조해 누락된 조건·권한 케이스·엣지 케이스를 지적한다.
- **이슈·버그 분석**: 재현 화면의 명세상 "정상 동작"을 확인해 버그/사양 여부를 판별하고,
  영향 범위(연관 화면·권한)를 짚는다.

## 문서 구조 (공통)

각 문서는 대체로 다음 순서를 따른다: `개요`(경로/역할/진입경로/**권한**) → `검색`(필드·탭별
노출) → `목록`(컬럼) → `Actions`(트리거/플로우/최종동작/**실패·예외**) → `User Flow`(mermaid).
스크린샷 이미지는 이 스킬에 포함하지 않았으니, 화면 확인이 필요하면 Figma MCP를 사용한다.

## 다른 QA 스킬과의 연동

`create-tc`, `qa-test-cases`, `qa-review-ticket`, `create-bug` 등 Roouty 기능을 대상으로 하는
작업의 **첫 단계로** 이 스킬을 사용해 해당 명세를 먼저 로드한 뒤 진행한다.

# 테스트 케이스 / 체크리스트 작성 규칙

## 기본 원칙
- 모든 결과물은 한국어로 작성
- 사용자 관점에서 시나리오 작성
- 정상 케이스, 예외 케이스, 경계값 케이스를 모두 포함
- 우선순위 기준: 높음(핵심 기능) / 중간(주요 기능) / 낮음(부가 기능)

---

## 테스트 케이스 형식

### ID 규칙
- `TC-[기능명약어]-[순번]` 형식 (예: TC-LOGIN-001)

### 테스트 케이스 표

| TC ID | 테스트 항목 | 전제조건 | 테스트 단계 | 기대 결과 | 우선순위 |
|-------|------------|---------|------------|---------|---------|
| TC-XXX-001 | 항목명 | - 전제조건 | 1. 단계1<br>2. 단계2 | - 기대결과 | 높음 |

### 항목 정의
- **TC ID**: 고유 식별자
- **테스트 항목**: 무엇을 테스트하는지 한 줄 요약
- **전제조건**: 테스트 실행 전 갖춰져야 할 상태
- **테스트 단계**: 순서대로 번호 매긴 실행 절차
- **기대 결과**: 테스트 통과 기준
- **우선순위**: 높음 / 중간 / 낮음

---

## 체크리스트 형식

```
## [기능/화면명] 체크리스트

### 기능 검증
- [ ] 항목1
- [ ] 항목2

### UI/UX 검증
- [ ] 항목1
- [ ] 항목2

### 예외 처리
- [ ] 항목1
- [ ] 항목2
```

---

## 분석 관점

티켓 또는 Figma가 주어졌을 때 아래 관점을 모두 검토:

1. **기능 동작**: 정상 흐름이 요구사항대로 동작하는가
2. **예외/오류 처리**: 잘못된 입력, 네트워크 오류, 빈 값 등
3. **경계값**: 최솟값, 최댓값, 한계 조건
4. **UI/UX**: 버튼 상태, 로딩, 메시지 노출 여부
5. **권한/접근**: 로그인 상태, 권한별 접근 제어
6. **데이터 정합성**: 저장/수정/삭제 후 결과가 올바른가

## 결과물 출력 형식

- 출력 형식(TC 표 / 체크리스트)은 **사용자가 명시적으로 지정**한다
- 지정이 없으면 형식을 먼저 물어본다
- 입력 소스(Jira, Figma 등)와 출력 형식은 무관하다 — 어떤 소스든 TC로도, 체크리스트로도 작성 가능

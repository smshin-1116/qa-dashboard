# 테스트 케이스 / 체크리스트 작성 규칙

## 기본 원칙
- 모든 결과물은 한국어로 작성
- 사용자 관점에서 시나리오 작성
- 정상 케이스, 예외 케이스, 경계값 케이스를 모두 포함

---

## 테스트 케이스 형식

### 기준: Sprint-22 표준 형식 (xlsx 및 채팅 미리보기 공통)

#### ID 규칙
- 숫자 순번 (1, 2, 3 …) — xlsx에서는 `=ROW()-{header_offset}` 수식 사용
- 별도 prefix(TC-LOGIN-001 등) 사용 금지

#### 컬럼 구성

| TC-ID | 1depth | 2depth | 전제조건 | 테스트 스텝 | 기대결과 | Comment |
|-------|--------|--------|---------|------------|---------|---------|
| 1 | 로그인 | 정상 로그인 | 유효한 계정 보유 | 1. 아이디 입력<br>2. 비밀번호 입력<br>3. 로그인 버튼 탭 | 메인 화면으로 이동 | |

#### 컬럼 정의
- **TC-ID**: 숫자 순번
- **1depth**: 기능 영역 대분류 (예: 로그인, 배차, 모니터링)
- **2depth**: 세부 기능 또는 화면 단위 (예: 정상 로그인, 비밀번호 오류, 세션 만료)
- **전제조건**: 테스트 실행 전 갖춰져야 할 상태
- **테스트 스텝**: 순서대로 번호 매긴 실행 절차
- **기대결과**: 테스트 통과 기준 (사용자가 보는 화면/동작 기준으로 기술)
- **Comment**: 특이사항, 주의점 (⚠ 핵심 버그 재현 케이스, ★ 신규 정책 케이스)

#### xlsx 전용 추가 컬럼 (채팅 미리보기에서는 생략)
- **Priority**: 빈칸 유지 (입력하지 않음)
- **Result(Android)** / **Result(iOS)**: 빈칸 유지
- **BTS 키**: 빈칸 유지

## 체크리스트 형식

### 작성 기준
- 항목 하나는 **검증 가능한 단일 행동**으로 작성한다
  - ❌ `버튼이 잘 동작하는가` (너무 포괄적)
  - ✅ `[저장] 버튼 탭 시 성공 토스트 노출되는가`
- 코드 레퍼런스(함수명, 상태값 등) 금지 — 사용자가 보는 화면/동작 기준으로 기술
- 정상 흐름과 예외 흐름을 모두 포함
- 카테고리는 아래 6개 기준으로 구성하되, 해당 없는 카테고리는 생략 가능

---

### 체크리스트 템플릿

```
## [기능/화면명] 체크리스트

### 기능 동작
- [ ] 정상 입력 시 기대 결과대로 동작하는가
- [ ] 저장/완료 후 목록/화면에 즉시 반영되는가
- [ ] 상태 전환(예: 대기 → 진행 중 → 완료)이 순서대로 동작하는가

### UI/UX
- [ ] 로딩 중 스피너 또는 스켈레톤이 노출되는가
- [ ] 빈 상태(데이터 없음)일 때 안내 문구가 표시되는가
- [ ] 버튼/입력 필드의 활성화/비활성화 상태가 상황에 맞게 전환되는가
- [ ] 성공/실패 시 토스트 또는 알림 메시지가 노출되는가

### 예외/오류 처리
- [ ] 필수 입력값 누락 시 오류 메시지가 노출되는가
- [ ] 네트워크 오류 시 에러 화면 또는 재시도 UI가 노출되는가
- [ ] 잘못된 값(경계값 초과, 특수문자 등) 입력 시 처리되는가

### 권한/접근 제어
- [ ] 비로그인 상태에서 접근 시 로그인 화면으로 이동하는가
- [ ] 권한 없는 사용자가 해당 기능에 접근할 수 없는가
- [ ] 권한에 따라 UI 요소(버튼, 메뉴 등)가 노출/숨김 처리되는가

### 데이터 정합성
- [ ] 저장 후 상세 화면에서 입력한 값이 그대로 표시되는가
- [ ] 수정 후 목록/연관 화면에 변경값이 반영되는가
- [ ] 삭제 후 해당 항목이 목록에서 제거되는가

### 성능/네트워크 (해당 기능에 실시간성·대용량 처리가 있는 경우)
- [ ] 목록 로딩 시간이 허용 범위 내인가 (기준: 3초 이내)
- [ ] 실시간 데이터(위치, 상태 등)가 지연 없이 갱신되는가
- [ ] 느린 네트워크 환경에서 앱이 비정상 종료되지 않는가
```

### wemeet 서비스 공통 체크 패턴

배차/모니터링/드라이버 앱에서 반복적으로 등장하는 검증 포인트:

**배차 관련**
- [ ] 배차 상태 전환(대기 → 배차 → 진행 → 완료)이 각 단계에서 정상 동작하는가
- [ ] 기사 앱 ↔ 모니터링 화면 간 배차 상태가 실시간으로 동기화되는가
- [ ] 배차 취소/변경 시 기사 앱에 즉시 반영되는가

**모니터링 관련**
- [ ] 주행 현황(위치, 진행률)이 실시간으로 갱신되는가
- [ ] 필터(날짜, 상태, 기사 등) 적용 후 목록이 정확하게 필터링되는가
- [ ] 대량 데이터(100건 이상) 조회 시 성능 저하 없이 로딩되는가

**공통 UX**
- [ ] 뒤로 가기 후 이전 화면 상태(스크롤 위치, 필터 등)가 유지되는가
- [ ] 앱 백그라운드 전환 후 재진입 시 데이터가 최신 상태로 갱신되는가

## 분석 관점

티켓, Figma, GitHub 중 무엇이 주어지든 아래 관점을 모두 검토:

1. **기능 동작**: 정상 흐름이 요구사항대로 동작하는가
2. **예외/오류 처리**: 잘못된 입력, 네트워크 오류, 빈 값 등
3. **경계값**: 최솟값, 최댓값, 한계 조건
4. **UI/UX**: 버튼 상태, 로딩, 메시지 노출 여부
5. **권한/접근**: 로그인 상태, 권한별 접근 제어
6. **데이터 정합성**: 저장/수정/삭제 후 결과가 올바른가

### 모드 1 — 특정 커밋 / PR 분석

커밋 해시, PR 번호, 브랜치가 주어졌을 때:

- **변경 범위**: 어떤 파일/모듈이 수정되었는가
- **영향 범위**: 변경된 코드가 연관 기능에 미치는 사이드 이펙트
- **API 변경**: 요청/응답 스펙, 파라미터, 상태코드 변경 여부
- **DB/상태 변경**: 스키마, 마이그레이션, 상태값 변경 여부
- **회귀 위험**: 기존에 동작하던 기능이 깨질 가능성

→ 변경된 부분을 중심으로 **좁고 깊게** 분석

---

### 모드 2 — 레포 전체 분석

특정 커밋 없이 기능명, 모듈명, 화면명만 주어졌을 때:

- **서비스 구조 파악**: 주요 도메인, 모듈, API 엔드포인트 구성 파악
- **핵심 플로우 도출**: 해당 기능의 주요 사용자 흐름 및 비즈니스 로직 파악
- **검증 포인트 도출**: 테스트해야 할 기능 범위와 우선순위 정의
- **연관 모듈 파악**: 해당 기능과 연동되는 다른 모듈/서비스 식별

→ 코드 전반을 읽고 **넓고 구조적으로** 분석

## 결과물 출력 형식

- 출력 형식(TC 표 / 체크리스트)은 **사용자가 명시적으로 지정**한다
- 지정이 없으면 형식을 먼저 물어본다
- 입력 소스(Jira, Figma 등)와 출력 형식은 무관하다 — 어떤 소스든 TC로도, 체크리스트로도 작성 가능
